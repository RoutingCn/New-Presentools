# HTML 演示美学范式研究草案

## 1. 调研结论

现有开源项目已经证明 HTML/Markdown/组件化代码可以承担演示渲染，但它们大多仍然把目标放在“生成一组 slides”。我们的方向不同：目标不是把 PPT 自动化，而是把演示重新定义为一个可讨论、可追踪、可修订、可发布的 HTML 信息系统。

可参考项目分为三类：

- HTML 演示运行时：reveal.js、Slidev、Marp、remark、impress.js、Spectacle。
- AI 演示生成工具：Presenton、PreGenie、SlideCoder、slide-forge-llm 等。
- 技术出版/数据叙事系统：Quarto、Observable 风格文档、Markdown + 可执行代码体系。

这些项目给我们的启发是：HTML 的价值不只是“能像 PPT 一样翻页”，而是可以把内容结构、交互路径、视觉秩序、数据证据和版本记忆统一到同一个产物里。

## 2. 关键参考

### reveal.js

reveal.js 是最成熟的 HTML presentation framework。它支持嵌套页面、Markdown、自动动画、PDF 导出、演讲者备注、LaTeX、代码高亮和 API。它适合作为我们理解“HTML 演示运行时”的基础参考。

参考：https://github.com/hakimel/reveal.js

### Slidev

Slidev 面向开发者，核心价值是 Markdown + Vue 组件 + Vite 生态。它证明了演示可以成为一个可编程工程，而不是一个静态文件。它对我们最有价值的地方不是界面，而是“演示内容可以被组件化、主题化、工程化”。

参考：https://github.com/slidevjs/slidev

### Marp

Marp 是 Markdown presentation ecosystem，强调用纯 Markdown 写演示，并可转换成 HTML、PDF、PPTX、图片等格式。它提醒我们：对普通用户而言，输入格式要轻，展示格式要强。

参考：https://github.com/marp-team/marp

### Presenton

Presenton 是开源 AI presentation generator，支持自托管、BYOK、多模型提供商、模板、PPTX/PDF 导出、API 和 MCP。它接近“AI 生成演示工具”，但仍然偏向 PPT/模板生成，不是以 HTML 作为唯一终态。

参考：https://github.com/presenton/presenton

### PreGenie

PreGenie 是一个基于 Slidev 的多智能体演示生成研究框架，重点是“生成后再审查、再生成”。这对我们很重要：HTML 生成不应该是一次性输出，而应有视觉、内容、结构、证据的多轮检查。

参考：https://arxiv.org/abs/2505.21660

### SlideAgent / DynaSlide

SlideAgent 研究的是根据自然语言更新既有幻灯片，并保留布局与风格。这说明“修改动作”本身应成为系统的一等能力，而不是附属于生成结果的小功能。

参考：https://arxiv.org/abs/2604.17894

## 3. 我们不应照抄什么

不应照抄 PPT 逻辑。PPT 的核心对象是页面；我们的核心对象应是内容节点、证据节点、叙事路径、展示组件和锁定产物。

不应照抄模板站逻辑。模板站的美学常常是“先好看，再塞内容”，这会导致内容关系被视觉效果挤压。我们的美学应先尊重内容关系，再通过代码把关系变得清楚、有节奏、有说服力。

不应只做 Markdown 转 slides。Markdown 很适合输入和中间态，但最终展示应允许复杂布局、交互、跳转、数据图、局部放大、分层浏览和演讲模式。

## 4. HTML 演示的美学范式

### 4.1 信息结构优先

每个页面或展示段落都必须回答：

- 这一屏的中心判断是什么？
- 它与上一个判断的关系是什么？
- 观众应该先看哪里，再看哪里？
- 哪些内容是主线，哪些内容是旁证，哪些内容可折叠？

美不是装饰，而是关系被清楚看见。

### 4.2 组件不是装饰块，而是语义容器

未来 HTML 生成不应直接让模型“自由写页面”，而应提供一套语义组件：

- `Claim`：核心判断。
- `Evidence`：证据、数据、引用、案例。
- `Contrast`：对比关系。
- `Process`：阶段、流程、因果链。
- `Matrix`：二维比较。
- `Case`：案例叙述。
- `Quote`：观点或原文。
- `Decision`：结论、选择、行动。
- `Appendix`：可跳转的补充材料。

视觉 Agent 的任务不是随意排版，而是把内容节点映射到正确的语义组件。

### 4.3 节奏比酷炫重要

一份 HTML 演示应有三种节奏：

- 阅读节奏：标题、正文、证据、注释之间有稳定层级。
- 演讲节奏：每一步只暴露当前要讲清楚的信息。
- 探索节奏：观众或演讲者可以跳转到证据、附录、案例和定义。

因此动画和交互必须服务于“逐步显露关系”，而不是服务于炫技。

### 4.4 代码美学要有约束

生成 HTML 时应强制执行一组约束：

- 不使用无意义的大渐变、浮动装饰、堆叠卡片。
- 字体层级稳定，不用视口宽度直接缩放文字。
- 每屏只允许一个主视觉重心。
- 内容区、辅助区、证据区边界清楚。
- 移动端可以阅读，桌面端可以演讲。
- 颜色承担分类和强调，不把页面染成单一色系。
- 所有交互都有可见状态：当前段落、已展开、可跳转、已锁定。

### 4.5 展示不是单文件，而是可审计产物

最终应同时保存：

- 全量内容版：保留结构、逐字稿、资料、讨论、修改过程。
- 展示 HTML 版：只呈现演示所需内容。
- 锁定版：用于发布，不被自动修订误改。
- 记忆 md：记录每次生成、讨论、接受、拒绝、回退、锁定。

这正是我们和普通 AI PPT 生成器的关键区别。

## 5. 对当前产品的落地建议

### 第一阶段：建立 HTML 生成契约

不要让模型直接“生成漂亮网页”，而是先让内容 Agent 产出结构化展示稿：

- 主题判断。
- 叙事路径。
- 内容节点。
- 证据节点。
- 例子节点。
- 可折叠补充。
- 演讲者逐字稿。

然后视觉 Agent 根据结构化稿生成 HTML。逐字稿只作为下载和演讲辅助，不混入展示正文。

### 第二阶段：建立组件库

先做 8-12 个稳定组件，而不是无限自由排版。每个组件包含：

- 适用内容类型。
- 必填字段。
- 可选字段。
- 视觉规则。
- 交互规则。
- 失败时的降级规则。

这样 AI 生成会更稳定，审美也更容易一致。

### 第三阶段：建立视觉审查器

HTML 生成后不直接锁定，而是经过检查：

- 标题是否重复。
- 内容是否重复。
- 结构是否缺少承接。
- 逐字稿是否混入正文。
- 是否有空 section。
- 是否有文本溢出。
- 是否有过度装饰。
- 是否缺少返回、跳转、下载或锁定入口。

这部分可以先用规则检查，后续再用模型审查。

### 第四阶段：把“修改”变成操作链

每个意见都要触发一个明确动作：

- 修改某节点。
- 拆分某节点。
- 合并某节点。
- 删除某节点。
- 增加证据。
- 改写语气。
- 重排结构。
- 重新生成 HTML 预览。
- 接受或拒绝修订。

界面必须显示动作状态，而不是只显示“讨论意见已提交”。

## 6. 推荐技术路线

短期不引入 reveal.js/Slidev 作为运行时，继续使用我们自己的单文件 HTML 输出，但吸收它们的成熟能力：键盘导航、演讲模式、备注、目录、PDF 友好、嵌套结构。

中期建立内部 HTML 组件库，让模型只在组件约束内生成页面。

长期可以提供两个渲染目标：

- 自有 HTML：适合产品默认展示和高度定制。
- Slidev/reveal.js 兼容导出：适合开发者和生态复用。

## 7. 下一步建议

我建议下一步先做“HTML 生成契约 + 组件库雏形”。DeepSeek 负责内容、结构和修订，HTML 生成使用本地美学 Markdown 引擎，不再引入远程 HTML 生成模型。这样系统链路会更短，问题更容易定位。

最小闭环应是：

主题输入 -> 深度分析 -> 内容结构 -> 逐字稿下载 -> HTML 结构化预览 -> 视觉审查 -> 锁定 HTML -> 记录到记忆 md。

这个闭环完成后，再研究更复杂的模板、动画和交互。
